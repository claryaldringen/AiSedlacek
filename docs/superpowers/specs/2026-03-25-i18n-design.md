# Vícejazyčnost (i18n) — Design Spec

## Kontext

Aplikace AiSedlacek (čtečka historických rukopisů) má veškeré UI texty hardcoded v češtině. Chceme přidat podporu vícejazyčnosti s otevřenou architekturou pro libovolný počet jazyků. Začínáme s češtinou a angličtinou.

## Rozhodnutí

- **Knihovna**: next-intl (nativní podpora Next.js 15 App Router, Server + Client Components)
- **Routing**: URL prefix (`/cs/workspace`, `/en/workspace`)
- **Výchozí jazyk**: angličtina
- **Detekce**: `Accept-Language` header → fallback EN
- **Scope**: kompletní — UI, API chybové hlášky, email šablony
- **Podporované locale**: `['en', 'cs']` (rozšiřitelné)

## Architektura routingu

Celý App Router se přesune pod `[locale]/` segment. API routes zůstávají bez locale prefixu.

```
apps/web/app/
  [locale]/
    (public)/page.tsx
    login/page.tsx
    forgot-password/page.tsx
    reset-password/page.tsx
    verify-email/page.tsx
    view/[slug]/page.tsx
    view/[slug]/[pageId]/page.tsx
    workspace/page.tsx
    workspace/billing/page.tsx
    layout.tsx
    not-found.tsx
    error.tsx
  api/                        # BEZ locale prefixu
```

### Middleware

next-intl middleware zajistí:
1. Autodetekce jazyka z `Accept-Language` headeru při prvním příchodu
2. Redirect `/workspace` → `/en/workspace` (nebo `/cs/workspace`)
3. Fallback na angličtinu
4. API routes (`/api/*`), statické assety a `_next/` vyloučené z locale routingu
5. Auth kontrola (session cookie) zachována — integrována do stejného middleware

Middleware se přepíše na kombinaci next-intl locale handlingu + existující auth logiky:

```typescript
import createMiddleware from 'next-intl/middleware';
import { NextRequest, NextResponse } from 'next/server';
import { routing } from './i18n/routing';

const intlMiddleware = createMiddleware(routing);

const PUBLIC_PATHS = ['/', '/login', '/forgot-password', '/reset-password', '/verify-email', '/view'];

function isPublicRoute(pathname: string): boolean {
  // Strip locale prefix: /en/login → /login, /cs/view/abc → /view/abc
  const withoutLocale = pathname.replace(/^\/(en|cs)/, '') || '/';
  return PUBLIC_PATHS.some(
    (p) => withoutLocale === p || withoutLocale.startsWith(p + '/'),
  );
}

function getLocaleFromPath(pathname: string): string {
  const match = pathname.match(/^\/(en|cs)\b/);
  return match?.[1] ?? routing.defaultLocale;
}

export function middleware(req: NextRequest): NextResponse {
  // API routes — skip locale, keep auth bypass
  if (req.nextUrl.pathname.startsWith('/api/')) {
    return NextResponse.next();
  }

  // Run next-intl middleware (locale detection + redirect)
  const response = intlMiddleware(req);

  // If next-intl returned a redirect (e.g. /workspace → /en/workspace), return it directly
  if (!response.ok) {
    return response;
  }

  // Auth check for non-public routes (on the already locale-prefixed pathname)
  const pathname = req.nextUrl.pathname;
  if (!isPublicRoute(pathname)) {
    const hasSession =
      req.cookies.has('authjs.session-token') ||
      req.cookies.has('__Secure-authjs.session-token');
    if (!hasSession) {
      const locale = getLocaleFromPath(pathname);
      const loginUrl = new URL(`/${locale}/login`, req.nextUrl.origin);
      loginUrl.searchParams.set('callbackUrl', pathname);
      return NextResponse.redirect(loginUrl);
    }
  }

  return response;
}
```

Public routes (bez auth): `/`, `/login`, `/forgot-password`, `/reset-password`, `/verify-email`, `/view/*`.

### Existující sdílené linky

Stávající URLs bez locale prefixu (např. `/view/abc123`) budou automaticky přesměrovány next-intl middleware na `/en/view/abc123` (HTTP 307). Toto je akceptovatelné — sdílené linky budou fungovat, jen přidají jeden redirect. next-intl defaultně používá 307 (temporary redirect), což je správné — obsah existuje pod oběma locale variantami.

### API Routes

API routes přijmou locale z custom headeru `X-Locale` (nastavovaného frontend fetch wrapperem) nebo fallback na `en`:

```tsx
import { getTranslations } from 'next-intl/server';
import { routing } from '@/i18n/routing';

function getLocaleFromRequest(request: NextRequest): string {
  // 1. Explicit header from frontend
  const header = request.headers.get('x-locale');
  if (header && routing.locales.includes(header as any)) return header;
  // 2. Fallback
  return routing.defaultLocale;
}

export async function POST(request: NextRequest) {
  const locale = getLocaleFromRequest(request);
  const t = await getTranslations({ locale, namespace: 'errors' });
  return NextResponse.json({ error: t('invalidJson') }, { status: 400 });
}
```

Frontend bude posílat `X-Locale` header automaticky — jednoduchý fetch wrapper, který čte aktuální locale z URL nebo next-intl kontextu.

### Metadata

V Next.js 15 je `params` Promise:

```tsx
export async function generateMetadata({ params }: { params: Promise<{ locale: string }> }) {
  const { locale } = await params;
  const t = await getTranslations({ locale, namespace: 'metadata' });
  return {
    title: { default: t('title'), template: '%s — AiSedlacek' },
    description: t('description'),
    openGraph: { locale: locale === 'cs' ? 'cs_CZ' : 'en_US' },
  };
}
```


### Logika výběru locale a propojení s middleware

Middleware nemá přístup k DB, proto se User.locale preference propaguje přes cookie `NEXT_LOCALE`:

1. Při přihlášení (signIn callback v auth.ts): načti `user.locale` z DB, nastav cookie `NEXT_LOCALE`
2. Při změně preference: `PATCH /api/user/locale` uloží do DB + nastaví cookie `NEXT_LOCALE`
3. next-intl middleware automaticky čte cookie `NEXT_LOCALE` (built-in podpora)

Priorita výběru locale:
1. Cookie `NEXT_LOCALE` (nastavená z user preference) → použij
2. `Accept-Language` header → nejlepší shoda z podporovaných locale
3. Fallback → `en`

### API endpoint pro změnu locale

`PATCH /api/user/locale` — přihlášený uživatel si změní preferenci.

Request: `{ locale: string }` — validace proti `routing.locales` (jinak 400)
Response: `{ locale: string }` + Set-Cookie `NEXT_LOCALE={locale}`

### Přepínač jazyka

Viditelný v navigaci (pro přihlášené i nepřihlášené). Klik přepne locale:
- Pro přihlášené: volá `PATCH /api/user/locale` (ukládá do DB + cookie) + redirect na `/{newLocale}/...`
- Pro nepřihlášené: přímý redirect na `/{newLocale}/...` (next-intl si zapamatuje volbu v cookie)

## next-intl konfigurace

### `apps/web/i18n/routing.ts`

```typescript
import { defineRouting } from 'next-intl/routing';

export const routing = defineRouting({
  locales: ['en', 'cs'],
  defaultLocale: 'en',
});
```

### `apps/web/i18n/request.ts`

```typescript
import { getRequestConfig } from 'next-intl/server';
import { routing } from './routing';

export default getRequestConfig(async ({ requestLocale }) => {
  let locale = await requestLocale;
  if (!locale || !routing.locales.includes(locale as any)) {
    locale = routing.defaultLocale;
  }
  return {
    locale,
    messages: (await import(`../messages/${locale}.json`)).default,
  };
});
```

### `next.config.ts` — přidat next-intl plugin

```typescript
import createNextIntlPlugin from 'next-intl/plugin';

const withNextIntl = createNextIntlPlugin('./i18n/request.ts');

export default withNextIntl({
  // ... existující konfigurace
});
```

## Error pages (not-found, error)

`not-found.tsx` a `error.tsx` pod `[locale]/` mají přístup k locale parametru a mohou používat `useTranslations()`.

Root-level `not-found.tsx` (pro zcela neplatné cesty, které neodpovídají žádné route) nemá locale kontext. Řešení: zobrazí anglický fallback text ("Page not found") bez next-intl. Toto je edge case — většina 404 chyb půjde přes `[locale]/not-found.tsx`.
